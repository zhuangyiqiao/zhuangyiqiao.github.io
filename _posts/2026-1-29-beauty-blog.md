# 我的 GitHub 博客搭建日记（二）：从模板到专属博客，手把手踩坑 & 改造

很多新手（包括我）第一次用别人的博客模板，都会发愁 “怎么把别人的东西换成自己的？” 今天我就把自己改造 Hux 模板的全过程拆成**简单步骤 + 避坑指南**，新手跟着做就能搞定～

## 一、第一步：先换 “看得见” 的东西 —— 图片和文字

### 1. 替换头像、背景图（5 分钟搞定）

模板里的所有图片都统一存在`img`文件夹中，直接替换即可完成视觉层的改造，步骤超简单：

- 找到`img`里的原作者头像`avatar-hux.jpg`，删掉后将自己的头像重命名（比如`avatar-cynthia.jpeg`）放进该文件夹；
- 文章默认背景图`img/post-bg-2015.jpg`也可同步替换，后续写新文章时能直接复用。

### 2. 改博客的 “招牌信息”—— 全局配置文件

根目录的`_config.yml`是博客的**总设置中心**，博客的标题、头像、社交链接等所有全局信息都从这里读取，打开后按注释替换以下关键信息即可：

yaml

```
# 博客名字（首页大标题）
title: 你的博客名（比如“Cynthia的成长笔记”）
# 搜索时显示的标题
SEOTitle: Cynthia的成长笔记 | 记录学习日常
# 博客简介（侧边栏会显示）
description: "分享我的学习和生活日常~"

# 你的社交账号（填自己的，不用的可以直接删行）
github_username: 你的GitHub用户名
weibo_username: 你的微博名

# 侧边栏的头像（填刚才放进img的头像名，路径要以/开头）
sidebar-avatar: /img/avatar-cynthia.jpeg
```

改完保存后，博客的核心信息就会全部变成你的，这一步是 “一键换身份” 的关键～

### 3. 写你的第一篇文章（终于有自己的内容啦）

模板里的示例文章都集中在`_posts`文件夹，先把这些示例全部删掉，再按规则创建自己的第一篇文章：

- 新建文件命名规则：**年 - 月 - 日 - 文章标题.md**（比如`2026-01-29-我的第一篇博客.md`），重点注意**月份和日期必须写两位**（1 月写 01、9 日写 09），否则博客无法识别；
- 文章头文件模板（直接复制，替换括号内内容即可）：

markdown

```
---
layout:     post
title:      "我的第一篇博客"（文章标题）
subtitle:   "终于有自己的博客啦"（副标题，可选）
date:       2026-01-29 12:00:00（发布时间）
author:     "Cynthia"（你的名字）
header-img: "img/你刚才换的背景图.jpg"（文章顶部背景图）
tags:
    - 博客搭建（文章标签，方便分类）
---

这里写你的文章内容~比如：“大家好，这是我用GitHub博客写的第一篇笔记！”
```

保存后，刷新博客首页就能看到自己的第一篇文章啦～

## 二、重点必看：修改 _config.yml 配置的核心注意事项

改完配置后图片不显示、信息不生效，90% 的问题都出在这部分，新手一定要逐条检查，避免踩坑！

### 1. 检查图片文件的「位置和命名」（最核心）

图片的路径和命名是最容易出错的地方，GitHub 对路径的识别严格遵循 “大小写、后缀完全匹配” 原则：

- ✅ 确认文件位置：图片必须放在和`_config.yml`同级的`img`文件夹中，目录结构如下：

plaintext

```
你的博客仓库/
├── _config.yml       # 配置文件
├── img/              # 图片文件夹（必须和配置文件同级）
│   └── avatar-cynthia.jpeg # 你的头像文件
└── ...其他文件
```

- ✅ 确认文件名完全匹配：
    
    1. 区分大小写：配置里写`avatar-cynthia.jpeg`，文件不能是`Avatar-Cynthia.JPEG`（Linux/GitHub 对大小写敏感）；
    2. 后缀一致：`jpeg`和`jpg`是不同后缀，文件是`avatar-cynthia.jpg`，配置就不能写`avatar-cynthia.jpeg`；
    3. 无多余符号：文件名建议去掉空格、特殊符号，比如`home bg.jpeg`改成`home-bg.jpeg`，避免路径识别失败。
    

### 2. 检查是否把图片推送到 GitHub 了

本地有图片但博客不显示，大概率是没把图片同步到远程仓库！GitHub Pages 只能读取远程仓库的文件，本地文件不会自动同步：

bash

运行

```
# 进入博客本地目录后依次执行
git add img/          # 单独添加img文件夹的修改
git commit -m "添加/替换博客图片"
git push origin main  # 推送到GitHub远程仓库
```

💡 小提醒：**每次修改任何文件后，都要执行`git add .`（添加所有修改）→`git commit -m "备注"`→`git push origin main`**，少一步都会导致远程仓库没有你的修改，博客自然无法生效。

## 三、我踩过的坑：云编辑器的 “延迟陷阱”，建议用本地 IDE

今天删`_posts`里的示例文章时，踩了**GitHub Codespace（云代码空间）的操作延迟坑**：

我明明只操作`_posts`文件夹的示例文件，却因为云编辑器反应慢、卡顿，页面自动跳转到其他目录，差点误碰`index.html`、`about.html`这类核心 HTML 文件 —— 这些文件是博客的 “骨架”，一旦删除博客就会显示异常，绝对不能动！

### 避坑建议：优先用本地 IDE（比如 VS Code）+ Git 管理

新手直接用**本地 VS Code 配合 Git**操作更稳妥，云编辑器的延迟和卡顿对新手不友好，而本地操作完全无延迟，能清晰看到自己的操作目录，还能实时预览博客修改效果，步骤也很简单：

1. 先把博客仓库克隆到本地（仅第一次需要）：

bash

运行

```
git clone https://github.com/你的用户名/你的用户名.github.io.git
```

2. 本地修改完文件后，按常规 Git 命令提交推送即可（和上文一致）：

bash

运行

```
git add .
git commit -m "修改博客信息/发布新文章"
git push origin main
```

## 四、我的进阶探索：慢慢摸清模板的架构逻辑

目前我没急着修改复杂的代码，而是先通过读模板的`README.md`文档和文件注释，慢慢理解模板的架构，毕竟先懂结构再修改，能避免乱改出问题：

- 先从**能直接替换的内容**（图片、文字、配置信息）入手，熟悉模板的基础使用；
- 逐步了解各文件夹的作用：`_includes`存放侧边栏、页脚等可复用的页面 “零件”，`_layouts`控制首页、文章页的整体排版样式；
- 目前正在通过这些基础修改，加强对整个 Hux 模板架构的理解，为后续的个性化定制做准备。

## 五、新手总结：改造 Hux 模板的正确步骤

跟着这个顺序做，能高效完成模板私有化，几乎不会踩坑：

1. 换图片：替换`img`文件夹里的头像、背景图，记好文件名和后缀；
2. 改配置：在`_config.yml`里替换所有原作者信息，重点检查图片路径；
3. 清示例 + 写第一篇：删掉`_posts`里的示例文章，按规则创建自己的第一篇博客；
4. 推送到 GitHub：确认所有修改后，通过 Git 把本地文件同步到远程仓库；
5. 避坑关键：用本地 IDE 代替云编辑器，避免操作延迟误碰核心文件。
